# 实施计划 — Summon 机制 + Skill 统一

## 概述

将现有的 agent 机制统一为 pi 原生的 skill 机制，新增 summon tool 让 main LLM 可以把任务派给 Lisa（worker），由 Lisa 自动选择 skill 执行。

---

## 第一步：删除 lib/agents.ts，改用 pi API 获取 skill 列表

**目标**：不再自建目录扫描，直接从 `pi.getCommands()` 获取 skill 名字列表。

### 做法

- **删除 `lib/agents.ts`** — 整个文件不再需要
- skill 列表不在初始化时获取，而是在 `getArgumentCompletions` 回调中**实时查询** `pi.getCommands()`，因为用户按 Tab 时 pi 一定已经初始化完毕：

```typescript
// dd.ts / dc.ts 中
export function registerDdCommand(pi: ExtensionAPI) {
  pi.registerCommand('dd', {
    getArgumentCompletions: (prefix: string): AutocompleteItem[] | null => {
      const skillNames = pi.getCommands()
        .filter(c => c.source === 'skill')
        .map(c => c.name.replace('skill:', ''));
      const filtered = skillNames.filter(s => s.startsWith(prefix));
      return filtered.length > 0
        ? filtered.map(s => ({ value: s, label: s }))
        : null;
    },
    handler: async (args, ctx) => { ... },
  });
}
```

- 不需要路径、不需要 `readSkill()`、不需要 `parseSkillInput()` —— 底层全由 pi 的 skill 机制处理
- 不需要在 `resources_discover` 中缓存列表（避免了时序问题）

## 第二步：协议层 msg-protocol.ts（删除 agent 字段）

**目标**：删除协议中的 `<agent>` 字段。skill 选择完全通过 markdown 文本驱动，不需要协议字段。

- `/dd coder xxxxx` → prompt 写 "使用 coder skill 执行以下任务：xxxxx"
- summon → prompt 写 "你必须使用合适的 skill 来执行这次任务"

### 改动点

- `PiMessage` 接口：删除 `agent` 字段
- `parseMessage()`：删除 `agent` 提取逻辑
- `buildMessage()`：删除 `agent` 参数，输出中不再有 `<agent: ...>` 行
- `generateCommId()` 不变

## 第四步：pane-comm/delegates.ts（删除 agent 引用 + 新增 summonMsgMaker）

**目标**：从 agent 切换到 skill，新增 summon 支持。

### 要改的

- **删除** `import { agentList, parseAgentInput }` — 不再需要
- **删除** `CompleteDelegateMsg` 中的 `agent` / `skill` 字段
- `delegateMsgMaker(userInput, skillNames)`：
  - 自己拆第一个词匹配 skillNames，提取 skill 名
  - **skill 名写进 markdown，不设协议字段**
  - markdown = skill 名存在时 ? `使用 ${skillName} skill 执行以下任务：\n\n${task}` : task
- `sendDelegate()`：
  - 调用 `buildMessage()` 时不再传 `agent` 参数（因为第二步已经删了）
  - 其他逻辑不变（创建 pane、轮询就绪、发送）

### 新增

- `summonMsgMaker(task: string)`：类似 `delegateMsgMaker`，但 commType 为 `'Summon'`，不解析 skill 名，整个 task 作为 markdown

## 第五步：pane-comm/dd.ts + dc.ts（补全源改为 skill）

**目标**：tab 补全和提示信息从 agent 改为 skill。

### 要改的（两个文件改动相同）

- **删除** `import { agentList }`
- `getArgumentCompletions` 改为通过 `pi.getCommands()` 实时获取 skill 列表（见第一步）
- 提示文案：`可用的 agents:` → `可用的 skills:`
- 用法提示：`/dd [agentName]` → `/dd [skillName]`
- `delegateMsgMaker()` 调用时传入 skillNames

## 第六步：pane-comm/interceptor.ts（简化 skill 处理 + Summon 支持）

**目标**：worker 侧删除手动 skill 加载逻辑，新增 Summon 类型处理。

### 要改的

- **删除** `import { readAgent }`
- **删除** agent/skill 的手动加载逻辑（`readAgent()` + `agentContent` 拼接 + `enhancedContent`）
- input hook 简化：**直接使用 `message.markdown`**，不需要拼接 enhancedContent
  - 当前代码：`agentContent ? agentContent + '\n---\n**任务:**\n' + markdown : markdown`
  - 新代码：直接用 `message.markdown`（里面已经包含 skill 指令）
- 删除通知文案 `已加载 agent:`

### Summon 类型处理

- `Delegate` 和 `Summon` 的处理逻辑完全一致（都是收到 markdown → 送给 LLM → 等回复 → 回复 + 关闭）
- 唯一区别是 prompt 前缀：`[来自 ${fromName} 的Delegate]` vs `[来自 ${fromName} 的Summon]`
- 不需要特殊分支

## 第七步：pane-comm/summon.ts（新增）

**目标**：注册 main LLM 的 summon tool。

### 实现

```typescript
export function registerSummon(pi: ExtensionAPI) {
  pi.registerTool('summon', {
    description: `将任务派给 ${workerName} 执行。${workerName} 会自动选择合适的 skill 来完成任务。`,
    parameters: {
      type: 'object',
      properties: {
        task: { type: 'string', description: '给 worker 的完整任务描述' }
      },
      required: ['task']
    },
    handler: async (args, ctx) => {
      const task = args.task;
      const config = loadConfig();

      const msg = summonMsgMaker(task);
      // prompt 末尾附加 skill 使用指令
      msg.markdown = task + '\n\n' + (config.summonPromptSuffix || '你必须使用合适的skill来执行这次任务。');
      await sendDelegate(ctx, msg);
      return { content: [{ type: 'text', text: '✓ 已将任务派给 ' + config.names.worker }] };
    }
  });
}
```

### 要点

- 只有一个参数 `task`
- 不需要 skill/agent 参数——Lisa 自己选
- prompt 末尾固定附加 skill 指令（从 config 读取）
- 复用 `sendDelegate()` 创建 pane + 发送

## 实施顺序

按依赖关系，从底层到上层：

```
1. 删除 lib/agents.ts         ← 不再需要
2. msg-protocol.ts            ← 删除 <agent> 字段
3. config.ts + config.json    ← 新增配置项
4. delegates.ts               ← 删除 agent 引用，新增 summonMsgMaker
5. dd.ts + dc.ts              ← 改用 pi.getCommands() 做补全
6. interceptor.ts             ← 删除 readAgent 逻辑，简化为直接用 markdown
7. summon.ts                  ← 新文件
8. index.ts                   ← 注册 summon
9. 文件清理                    ← 删除旧文件，更新 package.json
```

每一步改完后可独立验证，不需要全部改完才能测试。

## 风险点

1. **pi skill 触发可靠性**：依赖 pi 在 prompt 中明确提到 skill 名时能可靠加载。但 pi 的 skill 列表在 system prompt 中始终可见，且 prompt 中有明确指令，预期可靠性很高。
2. **向后兼容**：如果用户还在用 `agents/` 目录，升级后不再被发现。需要在 CHANGELOG 中明确说明迁移路径。
3. **pi.getCommands() 时序**：在 `getArgumentCompletions` 回调中实时调用，此时 pi 已完全初始化，不存在时序问题。
