## 最终方案

### 整体架构

```
用户:"让Lisa执行这个计划"
  → main LLM:根据上下文起草 prompt
    → summon tool:建 pane,发消息(附带所有扫描到的可用 agent 列表)
         agent 列表每项包含:名称、一句话介绍、文件全路径

Lisa(worker LLM):
  1. 读任务 + agent 列表
  2. 分析用哪个 agent 最合适
  3. loadAgent tool 加载对应的 .md
  4. 按照 agent 指导执行任务
  5. 返回结果
```

### 两层 LLM 各司其职

- **main LLM**:理解用户意图 + 整理上下文 + 写 prompt
- **worker LLM (Lisa)**:读任务 + 分析选 agent + 执行

### commType 区分任务来源

| 来源 | commType | worker 行为 |
|---|---|---|
| `/dd` `/dc` 命令 | `Delegate` | 原有流程，直接执行 |
| summon tool（Lisa） | `Summon` | 带 agent 列表，LLM 选 agent → loadAgent → 执行 |

### 具体改动

#### 1. main 侧:新增 `summon` tool

- 只有一个参数 `task`(main LLM 起草的 prompt),不需要 agent 参数
- 调用时复用现有 `delegateMsgMaker` + `sendDelegate`(dd 的流程)
- tool name 和 commType 都使用 `Summon`
- worker 的名字（如 Lisa）从 config.json 的 `names.worker` 读取，不写死
- 消息中附带所有扫描到的 agent 列表(名称 + 一句话介绍 + 全路径)
- 保留 `/dd` 和 `/dc` 命令作为快捷备用
- agent 列表通过协议新增的 `<agentList>` 标签传递，格式：
```xml
<agentList>
    [
      {"name": "coder", "description": "负责代码编写和修改", "path": "/path/to/coder.md"},
      {"name": "reviewer", "description": "负责代码审查", "path": "/path/to/reviewer.md"}
    ]
</agentList>
```

#### 2. worker 侧:新增 `loadAgent` tool

- Lisa 收到 commType 为 `Summon` 的任务后，看到附带的 agent 列表
- Lisa 根据任务分析选哪个 agent 最合适
- 调用 `loadAgent` 按文件全路径读取 .md 内容
- 按照 agent 内容的指导完成任务
- 如果没有合适的 agent,Lisa 自己干

#### 3. agent 列表

- main pi 发消息时附带所有扫描到的 agent 列表（名称 + 一句话介绍 + 全路径）
- worker pi 不需要自己扫描，直接用消息中附带的列表
- Lisa 看到列表后决定 loadAgent 哪一个

#### 4. agent 优先级

- 项目级 `.pi/agents/` > 全局 `~/.pi/agent/agents/`
- 同名 agent 就近覆盖（现有逻辑已支持）
