# Summon 机制 — 让 Lisa 按需选择 Skill 执行任务

## 核心思路

用户对 main LLM 说"让 Lisa 去做 XXX"，main LLM 根据对话上下文起草 prompt，通过 summon tool 发给 worker（Lisa）。Lisa 收到任务后，利用 pi 原生的 skill 机制自动选择合适的 skill 来执行。

### 两层 LLM 各司其职

- **main LLM**：理解用户意图 + 整理上下文 + 写 prompt（末尾附加 skill 使用指令）
- **worker LLM (Lisa)**：读任务 → pi 的 skill 系统自动匹配 → 加载并执行 → 返回结果

### 关键决策：Agent = Skill

不再维护独立的 `agents/` 目录和自建的扫描/加载机制。统一使用 pi 原生的 skill 机制：

- skill 文件放在 pi 的标准目录：`.pi/skills/`（项目级）、`~/.pi/agent/skills/`（全局）
- 底层走 pi 的 skill 发现、描述注入 system prompt、按需 read 的完整流程
- 不需要自建 `loadAgent` tool、不需要在协议中传 `<agentList>`

对用户来说，`/dd coder xxxxx` 的用法不变，只是底层从 "agent" 统一成了 "skill"。

---

## 整体架构

```
用户: "让Lisa执行这个计划"
  → main LLM: 根据上下文起草 prompt（末尾附加 skill 使用指令）
    → summon tool: 创建 worker pane，发送消息

Lisa (worker LLM):
  1. 收到 prompt（含 "你必须使用合适的skill来执行这次任务"）
  2. pi 的 skill 系统已在 system prompt 中注入了可用 skill 列表（name + description）
  3. Lisa 根据任务选择合适的 skill → read 加载 → 执行
  4. 完成后返回结果
```

---

## 三种任务来源

| 来源 | commType | 说明 |
|------|----------|------|
| `/dd skillName <task>` | `Delegate` | 用户指定 skill，不带上下文 |
| `/dc skillName <task>` | `Delegate` | 用户指定 skill，带上下文 |
| summon tool（Lisa） | `Summon` | main LLM 写 prompt，Lisa 自己选 skill |

## 具体改动

### 1. 统一为 Skill 机制

- **删除** `agents/` 目录扫描逻辑（`buildAgentList()`、`parseAgentInput()` 等）
- **改用** pi 原生 skill 目录：`.pi/skills/`、`~/.pi/agent/skills/`
- `/dd`、`/dc` 的第一个参数从 agent 名改为 skill 名
- tab 补全改为扫描 skill 目录

### 2. main 侧：新增 `summon` tool

- 只有一个参数 `task`（main LLM 起草的 prompt）
- 调用时复用现有 `delegateMsgMaker` + `sendDelegate`（`/dd` 的流程）
- tool name 和 commType 使用 `Summon`
- worker 名字从 `config.json` 的 `names.worker` 读取，不写死
- prompt 末尾固定附加：`你必须使用合适的skill来执行这次任务。`
- 保留 `/dd` 和 `/dc` 命令作为用户直接指定 skill 的快捷方式

### 3. worker 侧：无需改动

- Lisa 收到 commType 为 `Summon` 的任务后，正常处理
- pi 的 skill 系统已经将可用 skill 列表注入 system prompt
- Lisa 看到 prompt 中的 skill 使用指令，自然选择并加载合适的 skill
- 不需要 `loadAgent` tool、不需要协议中传 agent 列表

### 4. Skill 文件格式

沿用 pi 标准 skill 格式，与原 agent 文件的 frontmatter 完全兼容：

```
skills/
├── coder/
│   └── SKILL.md
├── code-review/
│   └── SKILL.md
├── scout/
│   └── SKILL.md
└── store-path/
    └── SKILL.md
```

或裸 `.md` 文件直接放在 `.pi/skills/` 下（pi 自动识别）：

```
.pi/skills/
├── coder.md
├── code-review.md
├── scout.md
└── store-path.md
```

### 5. 配置

`config.json` 新增 summon 相关配置（如有需要）：

```json
{
  "names": {
    "main": "Main",
    "worker": "Lisa"
  },
  "summonPromptSuffix": "你必须使用合适的skill来执行这次任务。"
}
```

## 改动量估算

| 改动 | 文件 | 说明 |
|------|------|------|
| 新增 summon tool | `pane-comm/summon.ts` | 复用 delegateMsgMaker + sendDelegate |
| 注册 summon tool | `index.ts` | 加一行 registerSummon(pi) |
| skill 扫描替代 agent 扫描 | `lib/agents.ts` | 改为扫描 skills 目录，或直接用 pi 的 skill 发现 |
| /dd /dc 参数名调整 | `pane-comm/dd.ts`、`dc.ts` | tab 补全改为 skill 列表 |
| commType Summon 支持 | `pane-comm/interceptor.ts` | input hook 中对 Summon 类型做处理（如果需要特殊逻辑） |

**不需要改的**：
- `msg-protocol.ts` — 不需要新增 `<agentList>` 标签
- `lib/zellij.ts` — 传输层不变
- `editor/editor.ts` — 无关
